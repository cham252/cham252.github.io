Hunt Methodology

Hypothesis
A user on the managed endpoint may have installed and used Tor Browser to establish anonymous outbound connections, bypassing corporate monitoring and potentially exfiltrating data over an undetected channel.

Three KQL queries, three MDE tables, one confirmed compromise — executed sequentially to build a complete attack timeline.

Stage 1
DeviceFileEvents
Detect installer write — was tor-browser-*.exe created on disk?
Stage 2
DeviceProcessEvents
Trace execution — did tor.exe actually run? What spawned it?
Stage 3
DeviceNetworkEvents
Confirm C2 — did tor.exe connect to a Tor relay node?
// 02

Attack Timeline

07:59 AM — DeviceFileEvents
Silent Tor Browser Installation
tor-browser-windows-x86_64-portable-14.5.8.exe /S written to disk at 07:59 AM. The /S flag confirms deliberate silent installation — no user prompts displayed.
07:59 AM – 08:09 AM — DeviceFileEvents
Browser Files Extracted
Tor Browser files extracted to user directory. Both tor.exe and firefox.exe (Tor Browser's bundled Firefox shell) detected as new file write events. Extraction path outside of standard install directory — consistent with portable execution.
08:17 AM — DeviceProcessEvents
tor.exe Process Created
Process chain: installer → firefox.exe (Tor shell) → tor.exe. Process ran for 7 minutes before making network connections — consistent with Tor circuit initialization and relay negotiation time.
08:24 AM — DeviceNetworkEvents
Live Outbound Connection to Tor Relay Confirmed
Outbound TCP from tor.exe to 140.235.237.13 on port 9001 — a publicly listed Tor relay node. Port 9001 is the standard Tor OR (onion routing) port. Corporate proxy was bypassed — traffic did not pass through monitored channels. Compromise confirmed.
// 03

KQL Queries Used

Three tables. One confirmed compromise.

// Stage 1 — Detect installer write (DeviceFileEvents) DeviceFileEvents | where FileName contains "tor" or FileName contains "firefox" | where ActionType == "FileCreated" | project Timestamp, DeviceName, FileName, FolderPath, InitiatingProcessFileName
// Stage 2 — Trace process execution (DeviceProcessEvents) DeviceProcessEvents | where FileName == "tor.exe" | project Timestamp, DeviceName, FileName, ProcessCommandLine, InitiatingProcessFileName, InitiatingProcessCommandLine
// Stage 3 — Confirm relay connection (DeviceNetworkEvents) DeviceNetworkEvents | where InitiatingProcessFileName == "tor.exe" | where RemotePort == 9001 | project Timestamp, DeviceName, RemoteIP, RemotePort, InitiatingProcessFileName, ActionType

Indicators of Compromise

IOC · File
tor-browser-windows-x86_64-portable-14.5.8.exe
Silent installer. /S flag confirms deliberate stealth installation.
IOC · Process
tor.exe
Tor relay process. Launched from user directory. Parent: firefox.exe (Tor Browser shell).
IOC · Remote IP
140.235.237.13
Confirmed public Tor relay node. Outbound TCP port 9001 (Tor OR protocol).
IOC · Port
TCP/9001
Standard Tor OR port. Connection bypassed corporate proxy — unmonitored exfiltration channel.
IOC · Process
firefox.exe (Tor Browser)
Tor-bundled Firefox shell. Not the legitimate Mozilla Firefox binary. Extracted alongside tor.exe.

// 05

Remediation Steps

1
Isolate the endpoint via MDE network isolation — prevent further exfiltration or lateral movement while investigation is active.
2
Terminate tor.exe and firefox.exe (Tor shell). Kill full process tree — not just the child process.
3
Delete all Tor files from extraction directories. Search recursively for tor.exe, tor-browser-*.exe, and the full browser folder.
4
Block Tor relay IPs and port 9001 at firewall and proxy. Add 140.235.237.13 and known Tor exit nodes to the block list.
5
Review 30 days of user activity — email, USB, file transfers, and outbound connections to Tor-adjacent infrastructure.
6
Escalate to HR and Legal if insider threat confirmed. Preserve evidence chain-of-custody for disciplinary or legal action.
7
Update detection rules — MDE custom detection for tor.exe, tor-browser-*.exe file creation, and port 9001 outbound connections.
