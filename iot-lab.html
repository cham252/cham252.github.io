Threat Model — STRIDE Analysis

Before writing a single line of remediation code, a formal STRIDE threat model was completed against the HYDROLOGIC water monitoring pipeline at Grand Marina hotel. Six threat categories were analyzed across sensors, MQTT broker, server, and dashboard components.

S — Spoofing
Rogue device could impersonate a legitimate water sensor and inject false telemetry into the pipeline. No device authentication existed in the insecure baseline.
T — Tampering
MQTT messages transmitted in plaintext — any on-path attacker could intercept and modify water flow or pressure readings before they reached the server.
R — Repudiation
No message signing or audit trail — malicious messages could be sent with no way to attribute them to a specific device or actor post-incident.
I — Info Disclosure
Unencrypted MQTT traffic exposed sensor telemetry (flow, pressure, anomaly states) in plaintext to anyone on the network segment.
D — Denial of Service
Open broker with no authentication allowed any device to flood topics with garbage data, disrupting monitoring and masking real leak or pressure events.
E — Elevation of Privilege
No topic-level access controls — any connected client could publish to any topic, including command channels intended only for authorized management systems.

Hardening Progression

Each phase built on the previous. The insecure baseline was documented first to establish a clear attack surface — then closed layer by layer with measurable outcomes at each step.

Phase 0 · Baseline
Insecure Pipeline
insecure-pipeline/
Plaintext MQTT over port 1883. No authentication — any client can connect to the broker and subscribe to or publish on any topic. No encryption, no signing, no access control. Water sensor telemetry fully exposed on the network.
No encryption No authentication Port 1883 (plaintext) Open broker
Phase 1
Secure Pipeline — TLS
secure-pipeline-TLS/
Migrated to TLS over port 8883. All MQTT traffic encrypted in transit — eliminates passive eavesdropping and man-in-the-middle interception of sensor telemetry. Mosquitto broker configured with CA certificate and server certificate. Clients verify server identity before connecting.
TLS encryption Port 8883 CA + Server cert Server authentication
Phase 2
Secure Pipeline — mTLS
secure-pipeline-MTLS/
Upgraded to mutual TLS — broker now requires each sensor to present its own client certificate during the handshake. Only pre-authorized devices with valid, CA-signed certificates can publish data. Rogue device spoofing attack vector fully closed. Measured 22ms→25ms latency impact at sustained 50 msg/sec throughput — operationally acceptable overhead.
Mutual TLS Client cert auth PKI per device 22→25ms latency
Phase 3
Replay Attack Protection
replay-attack-protection/
mTLS encrypts and authenticates the channel — but a captured legitimate message could still be replayed later. Added a three-layer replay defense: timestamp validation (messages older than N seconds rejected), monotonic counter (out-of-sequence messages dropped), and HMAC message signing with a shared secret (tampered messages rejected at broker level). All three layers active simultaneously.
Timestamp validation Monotonic counter HMAC signing Shared secret
