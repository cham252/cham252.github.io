// 01 Background & Objectives

Standard access control lists (ACLs) are router configuration scripts that permit or deny packets based solely on source IP address. This lab focused on defining filtering criteria, configuring standard ACLs, applying them to router interfaces, and verifying the implementation through structured connectivity tests. The three-router topology was pre-configured with IP addresses and EIGRP routing.

P1 Plan an ACL Implementation — analyze network policies, determine which traffic to filter, and choose interface placement based on best practices for standard ACLs.
P2 Configure, Apply, and Verify — write ACL statements, apply to correct interfaces with correct direction, verify with show commands and live connectivity tests.

// 03 Part 1 — Plan ACL Implementation

Before any configuration, full connectivity was confirmed by pinging all devices from each PC. All pings succeeded. Then the two network policies were analyzed to determine ACL design and placement.

→
Standard ACL Placement Rule: Standard ACLs filter on source IP only. Placing them near the source would block that traffic to all destinations. Place them outbound on the interface closest to the destination to limit blast radius.
// R2 Policy

R2 — WebServer Protection

192.168.11.0/24 (PC2) — denied access to WebServer at 192.168.20.254
All other traffic — permitted
Placement: G0/0 outbound on R2 (facing 192.168.20.0/24)
// R3 Policy

R3 — PC3 Network Isolation

192.168.10.0/24 (PC1) — denied access to 192.168.30.0/24 (PC3)
All other traffic — permitted
Placement: G0/0 outbound on R3 (facing 192.168.30.0/24)
// 04 Part 2 — Configure & Apply Standard ACLs
STEP 1 Configure and Apply ACL on R2

Created ACL 1 on R2 denying the 192.168.11.0/24 network. Added permit any to override the implicit deny-all. Verified with show access-lists before applying outbound on G0/0.

R2> enable R2# configure terminal ! Deny source network 192.168.11.0/24 — wildcard 0.0.0.255 R2(config)# access-list 1 deny 192.168.11.0 0.0.0.255 ! Override implicit deny-all — permit all other traffic R2(config)# access-list 1 permit any ! Verify before applying R2# show access-lists Standard IP access list 1 10 deny 192.168.11.0 0.0.0.255 20 permit any ! Apply outbound on G0/0 — traffic leaving toward WebServer LAN R2(config)# interface GigabitEthernet0/0 R2(config-if)# ip access-group 1 out
STEP 2 Configure and Apply ACL on R3

Created ACL 1 on R3 denying the 192.168.10.0/24 network (PC1). Same structure — deny specific source, permit any, apply outbound on the interface facing the destination LAN.

R3> enable R3# configure terminal ! Deny source network 192.168.10.0/24 — wildcard 0.0.0.255 R3(config)# access-list 1 deny 192.168.10.0 0.0.0.255 ! Permit all other traffic R3(config)# access-list 1 permit any ! Verify before applying R3# show access-lists Standard IP access list 1 10 deny 192.168.10.0 0.0.0.255 20 permit any ! Apply outbound on G0/0 — traffic leaving toward PC3 LAN R3(config)# interface GigabitEthernet0/0 R3(config-if)# ip access-group 1 out

// 07 Key Concepts Applied
⚠
Wildcard Masks: The inverse of a subnet mask. 255.255.255.0 → 0.0.0.255. A 0 bit means "must match"; a 1 bit means "any value." Used to define the range of IPs an ACL statement targets.
→
Implicit Deny All: Every ACL ends with an invisible deny any. Without an explicit permit any, all unmatched traffic is dropped — a frequent misconfiguration that cuts off legitimate traffic.
✓
in vs. out: ip access-group 1 out filters packets as they leave the interface toward the destination LAN. in would filter packets entering from that LAN — a different enforcement point entirely.
